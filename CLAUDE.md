## Zennの基本的なフォーマット

### フロントマター（Front Matter）

すべての記事は以下のようなYAMLフロントマターで始まります：

```yaml
---
title: "記事のタイトル"
emoji: "😎"
type: "tech" # tech: 技術記事 / idea: アイデア
topics: [raycast, ai, llm]
published: true
publication_name: raycast_jp  # パブリケーション名（オプション）
published_at: 2024-12-02 11:45  # 公開日時（オプション）
---
```

**あなたの使い方の特徴：**
- `type`は`tech`（技術記事）と`idea`（アイデア）を使い分け
- 技術解説や開発ツールの紹介は`tech`
- プロダクト紹介やイベントレポートは`idea`または`tech`
- `topics`は2〜5個程度、関連性の高いキーワードを選択
- `publication_name`で`raycast_jp`や`singularity`などのパブリケーションに投稿

## あなたの執筆スタイルの特徴

### 1. 導入部の書き方

**パターンA: イベントレポート・プロダクトリリース**
```markdown
# 🎉 MulmoCast v1.0.5 リリース！

マルモキャスト バージョン1.0.5をリリースしました！このバージョンでは、5つの主な更新があります。
```

**パターンB: 技術解説・チュートリアル**
```markdown
## はじめに

この記事は、**Google Apps Scriptを少し触ったことがある方**を対象にしています。

Gmail自動化スクリプトを作成中に「**同じ件名のメールが複数あるのに、1通しか処理されない**」という問題に遭遇しました。
```

**パターンC: 情報整理系**
```markdown
ちゃんと理解せずに npm / yarn 等を実施していたので生成AI を活用して整理したものです。記事の最後に Perplexity Sonar Pro によるファクトチェックを入れております。
```

**冒頭部分の基本構成（全記事共通）：**

1. `:::message` ブロックを先頭に配置し、AI活用の旨を明記
   - 例: `筆者の実体験をもとに Claude を活用して整理しました。`
2. 次に1〜2段落の導入文（見出しなし、または `## はじめに`）
   - **見出し「## この記事について」は使わない**（冗長なため）
3. パターン別の導入文（上記パターンA/B/Cを参照）

**共通の特徴：**
- 読者対象を明確にする
- 問題や目的を最初に提示
- AI活用を素直に開示（Claude、Perplexityなど）
- 読者の興味を引く具体的な課題や成果を冒頭で提示

### 2. 記事の構成スタイル

#### イベントレポート型
```markdown
1. イベントの目的・成果
2. イベントの様子
   2.1. アーカイブ動画 / Toggeter
   2.2. 基調講演
   2.3. パネルディスカッション
   2.4. 登壇者様の発表
3. 謝辞
4. 次回イベントに向けて
```

#### プロダクトリリース型
```markdown
## 1. クリックで開くファイル選択画面でファイルを読み込み
## 2. プロジェクト内の生成済み画像・動画を他のBEATで再利用可能
## 3. 画像生成プロンプトでBEAT時間を指定可能（上級モード）
...
```
- 機能を番号付きで列挙
- 各機能に画像を添付
- シンプルで読みやすい構成

#### 技術解説型
```markdown
## 完成したシステムの概要
（コードブロック）

## スレッド処理の落とし穴と解決法
### 問題の発見
### 原因の調査
### 解決策

## 実装時のベストプラクティス
### 1. エラーハンドリング
### 2. 重複処理防止
### 3. デバッグ機能の充実

## まとめ
```

#### 比較・整理型
```markdown
## 1. npm と npx の違い
### 1.1. npm (Node Package Manager)
### 1.2. npx (Node Package Execute)
### 1.3. npx の動作の仕組み

## 2. npm のプロジェクト管理
...

## ファクトチェック by Perplexity Sonar Pro
（表形式で検証結果を提示）
```

## 文章表現の癖

### 1. 丁寧で親しみやすい語り口

**特徴的な表現：**
- 「〜していただけると幸いです」
- 「〜について詳しく解説します」
- 「お楽しみください！」
- 「みなさん奮って参加ください」
- 「〜という問題に遭遇しました」
- 「〜が判明した事実」

### 2. 読者に寄り添う表現

```markdown
:::message
筆者の実体験をもとに Claude 4 Sonnet を活用して再編集しました。
:::
```

```markdown
:::message
この分析は実験的なものであり、より詳細な検証が必要です。
:::
```

- `:::message`ブロックを活用して注意事項や補足を丁寧に記載
- AIを活用した場合は明記する姿勢
- 読者への配慮が随所に見られる

### 3. 結論を先に示すスタイル

- 「このバージョンでは、5つの主な更新があります」→その後に詳細
- 「今回はこの世界モデルベンチーマークを利用して、3つの比較を行いました」→リストで提示
- 結論ファーストで読みやすさを重視

## 技術的な記述スタイル

### 1. コードブロックの使い方

**良い例・悪い例を対比：**
```javascript
// ❌ 問題のあるコード
for (var i = 0; i < threads.length; i++) {
  var message = messages[messages.length - 1]; // 最新のメッセージのみ取得
}
```

```javascript
// ✅ 解決後のコード
for (var i = 0; i < threads.length; i++) {
  var messages = threads[i].getMessages();
  for (var j = 0; j < messages.length; j++) {
    var message = messages[j];
  }
}
```

**特徴：**
- 絵文字（❌、✅、🔥）を使って視覚的に区別
- コメントで説明を補足
- 重要な箇所には`// 🔥 重要：`などのマーク

### 2. 表（テーブル）の活用

**比較表：**
```markdown
| 機能 | npm | yarn |
|------|-----|------|
| パッケージのインストール | `npm install` | `yarn` または `yarn install` |
| パッケージの追加 | `npm install パッケージ名` | `yarn add パッケージ名` |
```

**ファクトチェック表：**
```markdown
|元文章|チェック結果|説明|ソース|修正提案|
|---|---|---|---|---|
|npm は Node.js のパッケージ管理ツール|正確|npmの基本的な説明として正確です。| [^1] [^2]|変更不要|
```

### 3. 具体例の提示

```markdown
**基本的な使い方:**
```bash
# グローバルインストール
npm install -g パッケージ名

# ローカルインストール
npm install パッケージ名
```
```

- 実行可能なコマンド例を必ず記載
- コメントで各行の説明を追加
- 読者がすぐに試せる形式

## マークダウンの使い方

### 1. Zenn独自記法の活用

**メッセージブロック：**
```markdown
:::message
**コード内の PropertiesService について**

このコードでは、転送先メールアドレス(`EMAIL_TO`)や最終実行時間(`last_run`)を保存するために `PropertiesService` を利用しています。
:::
```

**埋め込み：**
```markdown
@[slideshare](1Y9U74QH5nb9nv)

https://www.youtube.com/live/sPTnyuO9OCA?feature=shared&t=113
```

**脚注：**
```markdown
GraphAI[^1]、生産性向上ツール「Raycast[^2]」

[^1]: GraphAI: https://github.com/receptron/graphai
[^2]: Raycast: https://www.raycast.com
```

### 2. 画像の挿入パターン

```markdown
![File Selection Dialog 1](/images/release_v1_0_5_script/1p.png)

![](/images/20241123-raycast/20241123-sponsors.png)
```

- `/images/`配下に記事ごとのフォルダを作成
- わかりやすいファイル名を使用（1p.png, 2p.png など）

### 3. リンクの記載方法

**外部リンク：**
```markdown
https://github.com/SingularitySociety/WorldModels

[MulmoCast公式サイト](https://mulmocast.com)
```

**内部リンク（Zenn内）：**
```markdown
https://zenn.dev/raycast_jp/articles/2024-08-22-raycast-japan-presentaion-material
```

**YouTube動画（タイムスタンプ付き）：**
```markdown
[YouTube Link](https://www.youtube.com/live/sPTnyuO9OCA?feature=shared&t=727)
```

### 4. 見出しの階層構造

```markdown
# 1. イベントの目的・成果
## 2. イベントの様子
### 2.1. アーカイブ動画 / Toggeter
### 2.2. 基調講演: 中島聡氏
#### 2.2.1. AGIの時代が目前に
#### 2.2.2. GraphAI について
```

**特徴：**
- `#`は記事タイトルレベル（あまり使わない）
- `##`をメインセクション
- `###`をサブセクション
- 番号付けで階層を明確化（2.2.1など）

## あなた特有の執筆パターン

### 1. 情報の透明性

- AIツールの活用を明記（Claude、Perplexityなど）
- 「筆者の実体験をもとに」と前置き
- ファクトチェックの結果を表で掲載
- 不確実な情報には「実験的なものであり、より詳細な検証が必要」と注記

### 2. 実用性重視

- 「今日から使える！」「〜の活用法」
- 「完成したシステムの概要」を先に提示
- 「ベストプラクティス」セクションを設ける
- コマンド例やコードを実行可能な形で提供

### 3. 体系的な整理

- 比較表を多用
- 「〜の違い」「〜の使い分け」というテーマ
- 番号付きリストで情報を構造化
- 目次を明示的に記載（イベントレポートなど）

### 4. 読者への配慮

```markdown
**まとめ（スライドより抜粋）**
- 検索機能: 任意のテキストで絞り込み
- Type機能: テキスト、ファイル、画像、リンク、色で絞り込み
```

- 重要なポイントを箇条書きでまとめる
- 「〜を対象にしています」と明記
- デバッグ方法や確認方法も丁寧に記載

### 5. コミュニティ志向

```markdown
# 3. 謝辞
本イベントは、多くの企業様のご支援により実現することができました。

# 4. 次回イベントに向けて
次回イベントの詳細は、connpass / 各種SNS にて随時お知らせいたします。
```

- イベントレポートでは必ず謝辞を記載
- スポンサー企業を画像付きで紹介
- SNSリンク、Slackリンクを掲載して参加を促す

## 記事ファイルの命名規則

- 記事ファイル名は `YYYY-MM-DD-記事スラッグ.md` の形式にする
- 例: `2026-02-07-claude-code-extensibility-memo.md`
- 日付は記事作成日または公開予定日を使用

## 記事更新時のルール

### 更新注記の書き方

記事を大幅に更新する場合、以下の形式で更新注記を追加すること：

```markdown
:::message alert
**YYYY-MM-DD 更新:** [情報提供者への感謝 + リンク]、更新内容の要約。
:::
```

### 公式ドキュメント引用時のルール

- 公式ドキュメントの原文を引用する場合、引用ブロック（`>`）を使用
- 引用元の URL を必ず併記（例: `> — [タイトル](URL)`）
- 翻訳・要約ではなく、原文をそのまま引用することを優先

### 比較表作成のルール

- 「同上」「同左」は参照先が明確な場合のみ使用
- 紛らわしい場合は繰り返し記述する
- 複雑な概念には補足テーブルを追加（例: 設定の組み合わせと挙動の一覧）

## チェックリスト

新しい記事を書く際の確認項目：

### フロントマター
- [ ] `title`は読者の興味を引くものか
- [ ] `emoji`は内容に合っているか
- [ ] `type`は`tech`/`idea`を適切に選択
- [ ] `topics`は2〜5個、関連性の高いものを選択
- [ ] `published`の設定は適切か
- [ ] `publication_name`は必要に応じて設定

### 構成
- [ ] 冒頭で記事の目的や対象読者を明記
- [ ] 見出しに番号を付けて階層を明確化
- [ ] コードブロックに説明コメントを追加
- [ ] 重要な箇所に絵文字（✅❌🔥）を使用
- [ ] 表を使って情報を整理
- [ ] 画像は`/images/`配下に適切に配置

### 文体
- [ ] 丁寧で親しみやすい語り口
- [ ] 結論ファーストの構成
- [ ] AI活用時は明記
- [ ] 不確実な情報には注記

### 最後に
- [ ] まとめセクションを設ける
- [ ] 参考リンクを掲載
- [ ] イベント記事の場合は謝辞を記載
- [ ] 次のアクション（参加方法、ダウンロードリンクなど）を明記
